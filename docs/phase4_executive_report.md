## How Guardrails Work

Every answer generated by the RAG pipeline passes through `validate_response()` before reaching the student. This function runs five independent checks — all five must pass for the response to be delivered.

### The Five Checks

| Check | What It Catches | How It Works |
|-------|----------------|--------------|
| `has_citations` | Answers that don't reference their sources | Regex scan for `[1]`, `[2]`, etc. in the response text |
| `within_scope` | Medical advice that could be dangerous | Matches against a prohibited phrase list ("self-diagnose", "stop taking your medication", etc.) |
| `not_empty` | Blank or trivially short responses | Verifies stripped response exceeds 10 characters |
| `source_grounded` | Hallucinated content not found in sources | Token overlap analysis — at least 30% of response tokens must appear in the retrieved source chunks |
| `no_hallucinated_citations` | Citations to non-existent sources | Extracts all `[N]` references via regex and verifies each N is within the range of actual sources provided |

### Source Grounding In Detail

Source grounding is the most important check. It prevents the LLM from generating plausible-sounding medical content that isn't actually supported by the retrieved sources.

```
Response: "Heart failure symptoms include edema [1] and fatigue [2]."
Source [1]: "Heart failure symptoms include edema and fatigue."
Source [2]: "Hypertension management includes ACE inhibitors."

Response tokens: {heart, failure, symptoms, include, edema, and, fatigue, ...}
Source tokens:   {heart, failure, symptoms, include, edema, fatigue, hypertension, ...}
Overlap:         {heart, failure, symptoms, include, edema, fatigue, and}

Overlap ratio: 7/9 = 78% → PASS (threshold: 30%)
```

A response that invents medical claims not found in any source will have low token overlap and be blocked.

### Integration With the RAG Chain

Guardrails are wired directly into `rag_chain.py`, called after LLM generation but before the response is returned. The function receives the raw reranker output (wrapped format with `{'doc': {'text': ...}}`) so it can access full source text for grounding analysis. The guardrails module has zero external dependencies — only stdlib `re` and `logging` — so it cannot fail due to API connectivity or model loading issues.

```
LLM Generation
       |
       v
  validate_response(response, reranked_sources)
       |
       +--- has_citations?        --> regex [N] scan
       +--- within_scope?         --> prohibited phrase check
       +--- not_empty?            --> length check
       +--- source_grounded?      --> 30% token overlap
       +--- no_hallucinated_citations? --> citation range check
       |
       v
  All pass? --> return response
  Any fail? --> return response with passed=False + failed checks logged
```

### Why This Matters for Medical Education

In medical education, a wrong answer doesn't just fail a test — it could shape how a future physician thinks about patient care. The guardrails ensure:

1. **Every claim is traceable** — citations link every answer back to specific NIH source material
2. **No dangerous advice slips through** — prohibited phrases like "self-diagnose" and "stop taking your medication" are caught before delivery
3. **Hallucination is measurable** — source grounding gives a quantitative metric for how much the answer relies on retrieved evidence vs. the LLM's parametric knowledge
4. **Failed responses are visible** — the `passed` flag and individual check results let educators audit system behavior

## How the At-Risk Prediction Model Works

The `AtRiskPipeline` identifies students at risk of underperformance using engagement and academic signals. This demonstrates predictive modeling in an educational context — the same pattern that VSTAR would use with real student data.

### Synthetic Dataset

Since real student data is FERPA-protected, we generate a realistic synthetic dataset of 500 medical students:

| Feature | Distribution | Rationale |
|---------|-------------|-----------|
| `quiz_avg` | Normal(75, 12) | Mean of 75% with realistic spread |
| `attendance_pct` | Beta(8, 2) × 100 | Right-skewed — most students attend regularly |
| `resource_views` | Poisson(15) | Count data for resource interactions |
| `forum_posts` | Poisson(3) | Count data for forum activity |
| `time_on_platform_hrs` | Gamma(5, 2) | Right-skewed time distribution |
| `prior_gpa` | Normal(3.2, 0.5), clipped 0-4 | Prior academic performance |

**Label:** `at_risk = 1` when `quiz_avg < 65` OR `attendance_pct < 70`. This captures students struggling academically, disengaged, or both.

### Feature Engineering

Three engineered features capture patterns that raw metrics miss:

| Feature | Formula | What It Captures |
|---------|---------|-----------------|
| `engagement_score` | 0.3 × resource_views + 0.3 × forum_posts + 0.4 × time_on_platform | Overall platform engagement in a single metric |
| `quiz_trend` | quiz_avg − mean(quiz_avg) | Whether a student is above or below the cohort average |
| `attendance_x_quiz` | attendance_pct × quiz_avg / 100 | Interaction between showing up and performing — a student with 90% attendance but 50% quizzes is different from one with 50% attendance and 90% quizzes |

### Model Architecture

```
Raw Features (6) + Engineered Features (3) = 9 input features
       |
       v
  StandardScaler (normalize to zero mean, unit variance)
       |
       v
  GradientBoostingClassifier
       n_estimators=100
       max_depth=4
       learning_rate=0.1
       |
       v
  5-Fold Cross-Validation → F1 score (mean ± std)
       |
       v
  MLflow Logging
       params: model_type, n_estimators, max_depth, n_features
       metrics: f1_mean, f1_std
       artifact: serialized sklearn pipeline
```

### SHAP Explanations

SHAP (SHapley Additive exPlanations) values make the model interpretable for advisors. Instead of a black-box "this student is at risk," SHAP shows exactly which factors contribute:

- **High quiz_avg impact** → "This student's quiz scores are the primary reason they're flagged"
- **Low attendance impact** → "This student's attendance is driving the risk prediction"
- **Negative engagement_score** → "Low platform engagement suggests disengagement"

This transforms the model from a classification tool into an intervention guide — advisors know which lever to pull for each student.

## How Monitoring Works

`QueryMetrics` is a lightweight, in-memory monitoring system that tracks pipeline health across requests.

### Interface

```python
metrics = QueryMetrics()

# Called after each query
metrics.record(latency=150.0, passed=True, empty=False)

# Called by monitoring dashboards / health checks
metrics.summary()
# → {
#     "total_queries": 3,
#     "guardrail_fail_rate": 0.33,
#     "p50_latency_ms": 150.0,
#     "p95_latency_ms": 240.0,
#     "empty_result_rate": 0.33,
# }
```

### Design Decisions

- **Bounded memory:** Uses a `deque(maxlen=1000)` for latencies — only the most recent 1000 requests are tracked, preventing unbounded memory growth
- **Percentile-based latency:** P50 and P95 using numpy, not averages — averages hide tail latency that affects user experience
- **Guardrail failure rate:** Tracks what percentage of responses fail validation, a leading indicator of content quality degradation
- **Empty result rate:** Tracks queries that return no results, indicating gaps in the knowledge base

## The Test Suite

17 tests across 4 files, all running with mocked external dependencies (no API keys, no model downloads, no Pinecone connections required).

### Test Architecture

```
tests/
  conftest.py              # Shared fixtures
  test_guardrails.py       # 6 tests — response validation
  test_retrieval.py        # 3 tests — cross-encoder reranker
  test_quality_predictor.py # 4 tests — retrieval quality regression
  test_api.py              # 4 tests — FastAPI endpoints
```

### Mocking Strategy

| Dependency | Mock Approach | Why |
|-----------|---------------|-----|
| CrossEncoder (HuggingFace) | `unittest.mock.patch` at class level | Avoids 80MB model download in CI |
| Pinecone API | Not imported in tests | Guardrails and quality predictor are self-contained |
| Anthropic API | `unittest.mock.patch` on pipeline global | API tests use a mock pipeline that returns canned results |
| FastAPI lifespan | Replaced with no-op async context manager | Avoids loading all models during test startup |
| Parquet data files | Not needed | Tests generate their own synthetic data |

### What Each Test File Validates

**test_guardrails.py** (6 tests)
- Prohibited phrases caught → `within_scope=False`
- Out-of-range citations caught → `no_hallucinated_citations=False`
- Valid response passes all checks → `passed=True`
- Empty response caught → `not_empty=False`
- Missing citations caught → `has_citations=False`
- Return type is always plain dict

**test_retrieval.py** (3 tests)
- Medical content reranked above irrelevant content
- Empty candidate list returns empty result
- Output structure has `doc` and `rerank_score` keys

**test_quality_predictor.py** (4 tests)
- Feature extraction returns shape `(1, 5)`
- Untrained model returns 0.5 (neutral default)
- Trained model predictions stay in `[0.0, 1.0]` range
- Relevant chunks score higher than irrelevant chunks

**test_api.py** (4 tests)
- `/health` returns 200 with status info
- `/ask` returns valid `QueryResponse` with all fields
- `/ask` rejects too-short questions (422 validation error)
- Validation response includes all 6 guardrail fields

# Interview Talking Points

## 1. Why Guardrails Are a Core Feature, Not an Afterthought

> "In medical education, a wrong answer could mislead a future physician. Guardrails aren't optional — they're a core feature. I implemented five independent checks: citation presence, scope enforcement, content length, source grounding via token overlap, and hallucinated citation detection. Each check is a separate function that can be tested, monitored, and extended independently. The source grounding check is the most important — it measures what percentage of the response's content actually comes from the retrieved NIH sources, giving us a quantitative hallucination metric."

## 2. Source Grounding as a Measurable Safety Metric

> "The 30% token overlap threshold for source grounding was a deliberate design choice. It's high enough to catch responses that are mostly LLM-generated speculation, but low enough to allow the model to paraphrase and synthesize. In production, I'd A/B test this threshold against human evaluation — too high and you get robotic, source-parroting answers; too low and hallucination sneaks through. The key insight is that grounding should be a continuous metric, not just a binary pass/fail."

## 3. Predictive Modeling for Early Intervention

> "The at-risk model demonstrates the full ML lifecycle: synthetic data generation, feature engineering, cross-validated training, MLflow experiment tracking, and SHAP explanations. The feature engineering is where the value is — raw quiz scores tell you a student is struggling, but the interaction feature `attendance × quiz` tells you something different. A student with 90% attendance and 50% quiz scores needs tutoring; a student with 50% attendance and 90% quiz scores needs a nudge to show up. SHAP makes these distinctions visible to advisors."

## 4. The Test Suite as a Production Gate

> "I built the test suite with a mocking strategy that mirrors how production tests should work. Every external dependency — Pinecone, Anthropic API, HuggingFace models — is mocked so the suite runs in under 5 seconds with zero network calls. This means tests can run in CI on every commit without API keys or GPU instances. The 17 tests cover the happy path, edge cases like empty responses, and security boundaries like prohibited medical advice."

## 5. Monitoring That Catches Degradation Early

> "QueryMetrics uses a bounded deque of the last 1000 requests to track P50 and P95 latency — not averages, because averages hide tail latency. The guardrail failure rate is a leading indicator: if it starts climbing, it means either the LLM is generating lower-quality responses or the source corpus has gaps for common queries. In production, I'd alert on guardrail_fail_rate exceeding 5% over a rolling 15-minute window."

## 6. Key Numbers

- **5 guardrail checks** run on every response — independent, testable, extensible
- **30% token overlap** threshold for source grounding verification
- **500 synthetic students** with 9 features for at-risk prediction
- **5-fold cross-validation** with F1 scoring, logged to MLflow
- **17 tests** across 4 files, all passing in under 5 seconds
- **0 external API calls** required to run the full test suite
- **16/16** phase validation agent checks passing
