## What Was Built

A safety and quality assurance layer that wraps the Phase 3 RAG pipeline, ensuring every generated answer passes five guardrail checks before reaching a student. Alongside this, a predictive model that identifies at-risk learners from engagement data, a retrieval quality predictor trained via cross-encoder distillation, a query monitoring system, and a 32-test pytest suite covering all components.

### Phase 4 Architecture

```
RAG Pipeline Output (from Phase 3)
       |
       v
  Guardrail Validation (5 checks)
       |
       +--- not_empty         → Answer has substance (>0 chars)
       +--- has_citations     → References [1], [2], etc.
       +--- within_scope      → No prohibited medical advice
       +--- source_grounded   → 30% token overlap per citation
       +--- no_hallucinated_citations → No [N] beyond source count
       |
       v
  Pass / Fail decision
       |

  (latency, pass/fail, empty response)
       |
       v
  API Response to student

  Synthetic Student Data (500 students, seed=42)
       |
       v
  AtRiskPipeline

       |

       |
       v

       |
       v

```

## How the Guardrails Work

Every response generated by the RAG pipeline passes through `validate_response()` before being returned to the student. Each check is independent — a failure in any single check fails the overall validation.

### The Five Checks

| Check | What It Catches | How It Works |
|-------|----------------|--------------|
| `not_empty` | Blank or whitespace-only responses | `bool(response.strip())` — catches cases where the LLM returns nothing |
| `has_citations` | Ungrounded answers with no source attribution | Regex extracts `[N]` patterns, verifies at least one is within the valid source range |
| `within_scope` | Dangerous medical advice | Scans for 8 prohibited phrases: "self-diagnose", "self-medicate", "stop taking", "start taking", "prescribe", "i recommend you take", "you should take", "discontinue your" |
| `source_grounded` | Answers fabricated from LLM knowledge instead of sources | For each citation `[N]`, extracts the citing sentence, computes token overlap with the referenced source chunk — must exceed 30% |
| `no_hallucinated_citations` | References to non-existent sources | If 5 sources were retrieved, `[6]` or `[7]` in the response triggers failure |

### Why Per-Citation Sentence Grounding

An earlier implementation compared the entire response against all sources. This was too lenient — a response could have one well-grounded paragraph and one completely fabricated paragraph and still pass. The current implementation extracts the specific sentence containing each `[N]` citation and checks that sentence against the referenced source. This catches localized hallucination.

**Example:**
```
Response: "Heart failure causes edema and fatigue [1]. Mars is the fourth planet [2]."
Source [1]: "Heart failure symptoms include edema fatigue and shortness of breath"
Source [2]: "Heart failure treatment includes ACE inhibitors"

[1] sentence overlap with Source [1]: ~60% → PASS
[2] sentence overlap with Source [2]: ~5% → FAIL (below 30% threshold)
Overall source_grounded: FAIL
```

### Why These Specific Prohibited Phrases

In medical education, the boundary between educational content and medical advice is critical. A student asking "What are the symptoms of diabetes?" should get an educational answer. But the system must never cross into prescriptive advice ("you should take metformin") or discourage professional care ("self-diagnose"). The 8 prohibited phrases cover the most common patterns where an LLM might inadvertently generate directive medical advice.

## How the At-Risk Predictor Works

### The Problem

Medical education programs need early warning systems to identify students who may be falling behind before they fail exams or clinical rotations. The `AtRiskPipeline` demonstrates how engagement data can drive these predictions.

### Synthetic Data Generation

The model trains on synthetic data (500 students, `np.random.seed(42)`) with 6 engagement features:

| Feature | Distribution | What It Measures |
|---------|-------------|-----------------|
| `quiz_avg` | Normal(0.7, 0.15) | Average quiz score (0–1) |
| `assignment_completion` | Normal(0.75, 0.2) | Fraction of assignments completed |
| `login_frequency` | Poisson(5) | Platform logins per week |
| `forum_posts` | Poisson(3) | Discussion contributions per week |
| `time_on_platform_min` | Normal(120, 40) | Weekly platform minutes |
| `content_diversity` | Uniform(0, 1) | Variety of content accessed |

Risk labels are generated from a weighted combination: 40% quiz performance + 30% assignment completion + 15% login frequency + 15% forum activity, with Gaussian noise to simulate real-world label uncertainty.

### Model Architecture

```
Input (6 features)
    |
    v
StandardScaler (z-score normalization)
    |
    v
GradientBoostingClassifier
  n_estimators=100, max_depth=4, learning_rate=0.1
    |
    v
Binary prediction: at_risk (0 or 1) + probability
```

### MLflow Integration

Training logs to MLflow automatically:
- **Parameters:** model type, hyperparameters, feature count, student count
- **Metrics:** 5-fold cross-validated F1 score (mean and std), train F1
- **Artifacts:** Serialized sklearn pipeline

### SHAP Explanations

The notebook (`05_predictive_model.ipynb`) generates SHAP summary plots showing which features drive at-risk predictions. This is critical for medical education stakeholders — an administrator needs to understand *why* a student is flagged, not just *that* they were flagged. SHAP values make the model actionable: "This student is at risk primarily because their quiz average dropped and assignment completion is low" rather than a black-box score.

## How the Quality Predictor Works

### The Problem

The cross-encoder reranker from Phase 3 provides excellent relevance scores but adds ~200ms of latency. For real-time quality estimation (e.g., deciding whether to show a confidence badge), we need a faster alternative.

### Cross-Encoder Distillation

The quality predictor is trained via knowledge distillation:

1. Take 500 eval queries
2. For each query, retrieve 20 chunks via Pinecone
3. Score each (query, chunk) pair with the cross-encoder → ~10,000 training examples
4. Train a GradientBoostingRegressor to predict the cross-encoder score from 5 lightweight features

### Features

| Feature | Source | What It Captures |
|---------|--------|-----------------|
| `cosine_similarity` | Pinecone | Dense embedding relevance |
| `bm25_score` | BM25 index | Keyword match strength |
| `token_overlap` | Computed | Direct word overlap between query and chunk |
| `chunk_length` | Computed | Longer chunks tend to have more relevant content |
| `qtype_match` | Metadata | Whether the chunk's question type matches the query intent |

### Why Distillation Over Direct Use

| Approach | Latency | Accuracy | Use Case |
|----------|---------|----------|----------|
| Cross-encoder (Phase 3) | ~200ms for 40 chunks | Excellent | Reranking during retrieval |
| Quality predictor (Phase 4) | <1ms | Good | Real-time confidence estimation |

The quality predictor gives ~80% of the cross-encoder's discrimination at <1% of the latency. It runs inside the API response loop to populate the `predicted_quality` field without adding measurable overhead.

## How Monitoring Works

The `QueryMetrics` dataclass tracks per-query metrics in memory:

```python
@dataclass
class QueryMetrics:
    _latencies: list[float]    # Response time in ms
    _passed: list[bool]        # Guardrail pass/fail
    _empty: list[bool]         # Empty response detection
```

The `summary()` method computes aggregate stats:

| Metric | What It Tracks | Why It Matters |
|--------|---------------|----------------|
| `total_queries` | Request volume | Load monitoring |
| `guardrail_fail_rate` | Fraction failing any check | Quality degradation alert |
| `p50_latency_ms` | Median response time | User experience |
| `avg_latency_ms` | Mean response time | Capacity planning |
| `empty_response_rate` | Blank responses | LLM failure detection |

In production, these metrics would feed into an alerting system — a spike in `guardrail_fail_rate` might indicate a prompt injection attempt or an LLM behavior change after a model update.

## Evaluation Script

`scripts/run_eval_queries.py` runs sample queries from both the eval and test sets through a local BM25 + Anthropic pipeline (no Pinecone or HuggingFace model downloads required). It measures:

- **Guardrail pass rate** per query and per question type
- **Answer overlap** with reference answers (token-level recall)
- **Latency** end-to-end
- **Per-qtype breakdown** showing which medical specialties the pipeline handles best

This provides an end-to-end integration test of the full pipeline: ingestion → retrieval → generation → validation.

# Interview Talking Points

## 1. Why Guardrails Are Non-Negotiable in Medical Education

> "In medical education, a hallucinated drug interaction or fabricated citation doesn't just give a wrong answer — it could shape how a future physician thinks about patient care. My guardrails run five independent checks on every response: is it non-empty, does it cite sources, is it within educational scope, is each citation grounded in the actual source text, and are there any references to sources that don't exist. A failure in any single check blocks the response. This is defense in depth — no single point of failure."

## 2. Per-Citation Sentence Grounding

> "My initial grounding check compared the whole response against all sources. That was too lenient — one fabricated paragraph could hide behind one well-grounded paragraph. I refactored to extract the specific sentence containing each citation and check that sentence against the referenced source. This catches localized hallucination. The 30% token overlap threshold was calibrated against the eval set to balance catching fabrication without flagging legitimate paraphrasing."

## 3. Cross-Encoder Distillation for Real-Time Quality

> "The cross-encoder gives excellent relevance scores but costs 200ms for 40 candidates. For real-time quality estimation — like showing a confidence badge on each answer — I distilled its knowledge into a GradientBoostingRegressor trained on five lightweight features. It gives roughly 80% of the cross-encoder's discrimination at under 1ms. This is a common pattern in production ML: train with the expensive model, serve with the cheap one."

## 4. SHAP for Stakeholder Trust

> "When I show a medical education administrator that a student is 'at risk,' the first question is 'why?' A black-box score isn't actionable. SHAP values decompose each prediction into per-feature contributions: 'This student is flagged primarily because their quiz average dropped 15% and they stopped posting in forums.' That turns a prediction into an intervention — the advisor knows exactly what to discuss."

## 5. Testing Strategy

> "I wrote 32 tests covering four layers: guardrails, API endpoints, the quality predictor, and the reranker. Every test runs offline — Pinecone, Groq, and Anthropic are all mocked. This means the test suite runs in CI without API keys or network access. I test both happy paths and edge cases: empty inputs, hallucinated citations, untrained models, whitespace-only responses."

## 6. Key Numbers

- **5 guardrail checks** on every response, running independently
- **30% token overlap** threshold for source grounding
- **8 prohibited phrases** blocked for medical safety
- **500 synthetic students** with 6 engagement features for at-risk prediction
- **~10,000 training examples** for quality predictor distillation (500 queries x 20 chunks)
- **32 tests** across 4 test files, all running offline
- **<1ms** quality prediction vs **~200ms** cross-encoder
